### ‚ùì What you value about TypeScript.

In my opinion I think its one of the best things that happened to javascript! I
love sketching things out in plain js, for example I always enjoyed the
accessibilty of jotting things down on the Developer Tools in the browser
console, or just in any node terminal.

But being able to add typing to JS is what empowers the language to be able to
build big applications that scale with security, avoiding bugs and also serving
as a kind of documentation for other developers building or interfacing with the
codebase.

Another valuable impact is that it can serve as a contract between different
layers across the stack like frontend and backend (we can also validate with zod
or openAPI), and also through the Database via typed ORMs

One thing I also like is that it can be applied gradually in a codebase, so even
if a project or feature started quick and dirty with plain js, we can gradually
update its files to use Typescript.

### ‚ùìAutomated tests in the frontend and backend. Are they needed? What do you prioritize? Describe your overall experience writing tests for the apps you‚Äôve built.

I‚Äôm a big fan of unit testing, specially in big projects, but I think there are
many considerations and priorities. My top priority would be to cover business
logic properly, specially in the backend (it can also help us structure the code
into small, testable functions). I would normally try to enforce a very thorough
coverage on the backend side, specially with unit tests. In my experience,
writing tests get easier once there is patterns in place to follow - sometimes,
it might be challenging to find the right way to mock, inject dependencies etc.
Once it is done for one test, I find it easy to just repeat the patterns around.

In general terms, I would advocate for strong unit testing on the backend, and
integration testing covering the main expected behaviours of the endpoints (we
did it with Postman at Notabene)

**About Front End testing:**

I also used jest in the Front End with React Testing Library, and I had mixed
feelings: I think it offers a lot of value when testing complex behaviours in a
component or library, but I also felt that sometimes the effort to make tests
work was very sustancial compared to the return.

So around the front-end, I try to be critical on which areas of the code I want
to unit test (maybe now this is changing with AI tools, which can help in
dealing with the tests scaffolding and reduce friction on the creation of
tests). In general terms, I find more valuable the higher level integration
tests in the front end, which abstract away from the inner components
implementation details and are closer to what the user sees (but they are also
longer and harder to build). At notabene we also used Cypress for smoke e2e
testing (I liked it, but I found it a bit unstable and slow, so we were using it
with basic cases instead of testing details and they were running on a daily
instead of every PR. Maybe it has improved or there are newer alternatives for
e2e)

### ‚ùìNon-technically, what are the most important factors for a software development project to succeed in both time and quality?

This is a very broad topic! I‚Äôll try to sketch some ideas that come to mind
right now, and I guess there are some many angles that might be specific to the
company, project or team.

I had one CTO that liked to say ‚ÄúSales are the department of promises, we are
the department of apologies üòÄ‚Äù

But in general:

- A reasonable quality/speed balance. This might depend on the project and the
  level of maturity (in an early startup or an experimental feature, we might
  just need to hack something that works, and if it does, improve it in a later
  iteration, while on a more mature product or a ‚Äúcore feature‚Äù, we might want
  to be very quality oriented, reviewing and testing heavily). In any case, the
  quality/speed balance should be aligned with the project needs.
- I like working in slices (like in Shape Up methodology), which is focusing on
  bulding flows that work first. It helps identifying and filtering the core
  important features and the nice-to-haves, being customer centric and
  activating the feedback loop earlier. It‚Äôs not necessary to follow
  particularly this methodology to benefit from this pattern!
- a few things about teammates:
  - should open to learning new things,
  - generous with helping out each other,
  - respectful of others work (avoiding micromanagement, trusting, empowering)
  - respecting commitments within the work methodology, offering feedback with
    radical candor if needed, also protecting each other from unreasonable
    demands from business and management while also having a company-wide
    high-level view on what is important and really ‚Äùmoves the needle‚Äù
  - open to reduce scope if needed, if the deadlines are endangered
- Non technical people:
  - Ideally clear communication,
  - avoid changes of scope and respecting the sprints or commitments (try to not
    add new items randomly in the middle of a sprint),
  - reasonable time appetites (don‚Äôt plan assuming the optimal outcome)
  - compassionate about the great misteries of software development
